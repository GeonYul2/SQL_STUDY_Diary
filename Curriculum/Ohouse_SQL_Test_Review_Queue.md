# 오늘의집 SQL Test Review Queue

이미 풀었지만 다시 봐야 하는 문제들을 **시험 직전용 복습 큐**로 재구성한 문서다.

목표는 단순 복기가 아니라:

1. 문제를 다시 읽고
2. **가상으로 SQL을 직접 작성하고**
3. 내 답안을 검토받고
4. 재작성 포인트를 반영하는 것

---

## 운영 방식

### 풀이 루프

1. 문제를 고른다.
2. 정답 파일을 바로 보지 않고 직접 SQL을 작성한다.
3. 나에게 아래 형식으로 보낸다.

```md
문제: [문제명]
내 풀이 의도:
```sql
-- 내가 작성한 SQL
```
헷갈린 점:
```

4. 그러면 나는 아래 기준으로 검토한다.
   - 정답 가능성
   - 출력 컬럼/정렬 조건 충족 여부
   - JOIN/집계/윈도우 함수 논리 오류
   - NULL/중복/분모 0 등 예외 처리
   - 더 간단한 풀이 가능 여부
   - 오늘의집/데이터 QA 관점으로 확장 가능한지

### 권장 규칙

- 먼저 **정답 없이 15~25분** 혼자 풀기
- 막히면 힌트만 요청하기
- 완전히 끝난 뒤에만 기존 풀이 파일 열기
- 리뷰 후에는 “왜 틀렸는지”를 한 줄로 남기기

## C. 오늘의집 포지션 연결 포인트

재풀이 시 아래 관점을 같이 체크하면 좋다.

- 결과를 만들기 전에 **데이터 단위**를 분명히 했는가?
- JOIN 때문에 중복 집계가 생기지 않는가?
- 비율 계산 시 분모 0 또는 NULL 처리가 필요한가?
- 최신 상태 / 최고값 / TOP N 추출에서 윈도우 함수가 더 안전하지 않은가?
- 이 쿼리를 데이터 품질 검증용으로 바꾼다면 어떤 sanity check를 붙일 수 있는가?

## E. 내가 해줄 수 있는 역할

네가 문제를 보고 직접 SQL을 써 오면, 나는 아래 3가지 모드로 도와줄 수 있다.

### 1. 힌트 모드
- 정답은 안 주고 방향만 제시
- JOIN 구조, 집계 기준, 윈도우 함수 여부만 힌트

### 2. 리뷰 모드
- 네 SQL을 채점하듯 검토
- 틀린 이유 / 위험 포인트 / 수정 포인트 설명

### 3. 면접 모드
- “왜 이렇게 풀었는지” 설명까지 점검
- 데이터 품질 관점 / 실무 확장 관점 질문 추가
